Шифр НИР: **Интеграл-КБО-МС-ГосНИИАС-ИСМПУ-2026**. Срок: **28.07.2026**, целевой УГТ 3.

Документ описывает переход от текущего прототипа (Jupyter [`main.ipynb`](main.ipynb)) к
модульной системе гибридного управления: классический PID-контур + нейросетевой планировщик
коэффициентов (**NPGS**, §10, обучается PPO), защитный контур (Shield) и, опционально в будущем,
физически-информированный обсервер (PINN/PD-PINN). Архитектура — по диаграмме
[`PIDNN.mmd`](PIDNN.mmd), требования — по ТЗ [`ТЗ_Интеграл-КБО-МС_ИСМПУ_итог_ф.pdf`](ТЗ_Интеграл-КБО-МС_ИСМПУ_итог_ф.pdf).

## 1. Ключевые принципы

1. **Классический контур — это plant, а не конкурент.** Нейросеть не заменяет PID и **не встраивает его
   передаточную функцию в себя** (это запрещено ТЗ — потому это не классический PIDNN, а планировщик
   коэффициентов NPGS, §10). Она лишь генерирует *мультипликативные поправки* к коэффициентам PID и веса
   влияния каналов. При тождественных поправках (α = 1, веса = базовые) система обязана бит-в-бит
   воспроизводить классическое поведение. Это инвариант, проверяемый тестом на каждом этапе.
2. **Разделение по динамике, а не по органам.** Два канала: **продольный** (реверс + тормоза, симметрично)
   и **латеральный** (руление + дифференциальное торможение + ассиметричная тяга). Управление — распределением степени влияния
   каналов. Отказ (например NWS) → повышение веса латерального канала, а не переобучение с нуля.
3. **Одна абстракция среды — два бэкенда.** Обучение идёт на X-Plane 12, поставка (ТЗ 5.1.4) — `.exe`
   поверх согласованного ПИВ на стенд Заказчика ([`ics_connector.py`](ics_connector.py)). Оба скрыты за
   единым интерфейсом `SimInterface`, чтобы актор/шилд/контур не зависели от транспорта.
4. **Безопасность вне сети.** Shield — детерминированная функция, не обучается, всегда активна, имеет
   fallback на чистый классический контур. Нейросеть не может выдать команду, минуя Shield.
5. **Обсервер — опция с зарезервированными швами.** PINN не строим сейчас, но заранее резервируем слоты
   в Observation Space, тип `ObserverEstimate` и слагаемое `L_phys`, чтобы включение было неинвазивным.
6. **Детерминизм при поставке.** Замороженные веса, фиксированная нормализация, greedy-инференс,
   без обучения в рантайме (диаграмма, блок 7).

## 5. Observation Space (полный вектор состояния) — реализовано (`envs/observation.py`)

Собирается в `envs/observation.py` из телеметрии (`SimInterface`), состояния классического контура и
(в будущем) оценок обсервера. **Все компоненты нормируются** фиксированными константами из
`agent/normalization.py` (единые для обучения и поставки — иначе детерминизм ломается).

Регуляторов **5**: `steer`, `brake_l`, `brake_r`, `rev_l`, `rev_r` (в целевой архитектуре mermaid есть
6-й — `flaps`; добавляется опционально, см. §10).

| Блок | Признак | Ед. | Диапазон (норм.) | Источник |
|---|---|---|---|---|
| **Геометрия** | cross_track_error | м | ±30 → /30 | tracker |
| | heading_error | град | ±45 → /45 | tracker.guidance |
| | distance_to_end | м | 0..L → /L | tracker |
| | lookahead | м | 10..120 → /120 | tracker |
| **Скорость** | ground_speed | м/с | 0..110 → /110 | телеметрия |
| | reference_speed | м/с | 0..110 → /110 | trajectory |
| | speed_error | м/с | ±50 → /50 | вычисл. |
| | longitudinal_acceleration | м/с² | ±5 → /5 | конечная разность v |
| **Управление** (последнее применённое) | brake_l, brake_r | — | [0,1] | ControlsState |
| | reverse_l, reverse_r | — | [0,1] | ControlsState |
| | rudder_cmd | — | [-1,1] | ControlsState |
| **PID × 5 регуляторов** | Kp, Ki, Kd | — | log-scale, норм. по базе | PID (динам.) |
| | integral | — | ±anti_windup → /aw | PID (динам.) |
| | derivative (filtered) | — | норм. по масштабу | PID (динам.) |
| | output | — | [min_out,max_out]→[-1,1] | PID (динам.) |
| | min_out, max_out, integral_decay, der_filter_tf | — | константы сценария | PID (статич.*) |
| **Отказы** | reverse_l_fault, reverse_r_fault, nws_fault | флаг | {0,1} | FailureState / сценарий |
| **Погода** | wind_along, wind_cross | м/с | ±20 → /20 | телеметрия/сценарий |
| | runway_friction μ | — | 0..1 | сценарий (или обсервер) |
| | rain / precip | — | 0..1 | сценарий |
| | visibility | м | 0..10000 → лог | сценарий |
| **(Опция) Обсервер** | μ̂, ŵ_cross, brake_eff, nws_eff, τ_delay | — | норм. | PINN (§10) |

\* Статические PID-параметры (`min_out/max_out/integral_decay/der_filter_tf`) в текущей схеме актор **не
меняет** (он модулирует только Kp/Ki/Kd) — они постоянны в пределах сценария. Их включение в obs
избыточно; оставляем как **опциональный флаг** `include_static_pid_features` (нужно, только если позже
Shield/актор начнут адаптировать и границы). По умолчанию исключаем ради снижения размерности.

**История и рекуррентность.** Диаграмма предусматривает shared encoder GRU/LSTM и temporal encoder
обсервера над окном `X_t … X_{t-n}`. Поэтому:
- `observation.py` возвращает **один кадр**; окно собирает кольцевой буфер в `rollout_env.py`
  (`obs_history`, глубина `n` в конфиге).
- Актор с рекуррентным encoder читает последовательность; для MLP-варианта — конкатенацию кадров.

**Нормализация — единый контракт.** `normalization.py` хранит `mean/scale` (или физические пределы) как
константы, сериализуемые вместе с весами при экспорте (§11). Никаких «плавающих» статистик на инференсе.

## 6. Action Space (действие актора) — реализовано (`envs/action.py`)

Актор выдаёт **мультипликативные поправки**, а не абсолютные величины (философия §1).

1. **Поправки к коэффициентам PID.** Для каждого регулятора — `(α_p, α_i, α_d)`, центрированы на 1.0,
   применяются как `Kp_eff = Kp_base · α_p` (аналогично Ki, Kd). Головы по диаграмме: Heading (steer),
   Brake (L+R), Reverse (L+R), [Flap]. Итого `5×3 = 15` (или `6×3 = 18` с закрылками) поправок.
2. **Веса влияния каналов.** Скаляры (или мягкое распределение) `w_lon`, `w_lat` — степень вклада выходов
   продольного и латерального каналов. Именно ими достигается перераспределение при отказах (NWS →
   `w_lat↑`). Реализуются как множители к выходам каналов перед микшированием в `ControlsState`.
3. Выход головы — параметры распределения политики (mean, log_std) для PPO; действие сэмплируется при
   обучении, берётся mean (greedy) при поставке.

Границы поправок (например α ∈ [0.5, 1.5]) — это **первый уровень Shield** (§8), а не свойство сети.

## 7. Абстракция симулятора (`envs/sim_interface.py`) — реализовано

Единый интерфейс, за которым скрыт транспорт. Размещён в `envs/` (не `io/`): зависит от
`WeatherState`/`FailureMode`/`Scenario`, поэтому находится выше слоя транспорта. `reset`/`step`
возвращают **сырую** `Telemetry` (нормализованный Observation собирается в Этапе 2, §5).

```
class SimInterface(ABC):
    def reset(self, scenario: Scenario) -> Telemetry
    def step(self, command: ControlsState) -> Telemetry        # шаг 20 Гц
    def read_telemetry(self) -> Telemetry
    def apply_weather(self, w: WeatherState) -> None            # ветер, μ, дождь, видимость
    def inject_failure(self, f: FailureMode) -> None
    def clear_failures(self) -> None
    def teleport_touchdown(self, setup: TouchdownSetup) -> None
    def pause(self, flag: bool) -> None
    def update(self, distance_m: float) -> None                 # переменное сцепление и т.п.
    def close(self) -> None
```

- **`XPlaneBackend`** (обучение): обёртка над `XPlaneConnectX`; асинхронная телеметрия через
  `subscribeDREFs`; телепорт/пауза для быстрого сброса эпизода; запись погоды/μ через DREF (какие именно
  writable-DREF-ы — уточнить по [`DataRefs.txt`](DataRefs.txt), раздел `sim/weather/…`; вынести в отдельную
  research-задачу Этапа 1).
- **`ICSBackend`** (поставка): обёртка над `ICSBenchConnector`; маппинг `ICSInputs → Observation` и
  `ControlsState → ICSOutputs`. Управление погодой/отказами на стенде задаёт Заказчик — на этом бэкенде
  `apply_weather`/`inject_failure` no-op или диагностика читается из `ICSInputs.Fault*`/`RunwayCondition`.

Актор, Shield, критик, env зависят только от `SimInterface`. Это выполняет ТЗ 5.1.4 (перенос на стенд без
переписывания логики).

## 8. Генератор сценариев (`envs/scenario_generator.py`)

Диаграмма, блок 0. Сэмплирует ситуации из **всего** пространства признаков — нужны и штатные, и «ужасные»
(для робастности PPO). Оси рандомизации:

- **ВПП/сцепление:** сухая / мокрая / лёд (μ), переменное μ по длине.
- **Ветер:** постоянный боковой, порывы (случайный процесс).
- **Отказы:** NWS, отказ реверса Л/П, асимметрия тяги, некорректная конфигурация шасси, комбинации.
- **Датчики:** гауссов шум, смещения, выпадения (для робастности и OOD-детектора Shield).
- **Начальные условия:** скорость касания, точка касания, смещение от оси, курс.
- **Domain randomization:** совместная выборка + учебный план (curriculum: от штатных к тяжёлым).

Каждый сценарий — сериализуемый `Scenario` (для воспроизводимости и приёмочной батареи Этапа 5).
Пресеты классических PID из ячеек ноутбука (`# default`, `# left reverse fail`, `# NWS fault`, …)
переносятся в `config/scenarios.py` и служат как (а) базой для поправок, (б) классическим бейзлайном для
сравнения в GUI.

## 9. Shield — защитный контур (`agent/shield.py`) — реализовано

Детерминированная функция между актором и контуром. Три уровня + fallback (диаграмма, блок 4).

1. **Уровень 1 — поправки.** Clip слишком больших `α` и весов каналов к допустимым границам.
2. **Уровень 2 — коэффициенты PID.** После применения поправок проверяем `Kp/Ki/Kd`: hard bounds,
   rate limit (ограничение скорости изменения между тактами), консистентность знаков и диапазонов, OOD
   (выход состояния/коэффициентов за обучающее распределение).
3. **Уровень 3 — Runtime Safety.** Поведенческие проверки: слишком быстрое изменение коэффициента,
   чрезмерно агрессивные тормоза, реверс на низкой скорости (<60 узлов, ср. текущую логику канала),
   срыв по курсу и т.п.
4. **Projection to Safe Set / Fallback.** При нарушении — проекция в безопасное множество; при OOD/грубом
   нарушении — переключение на **классический PID** (или автопилот).

Связь с loss: активация любого уровня → штраф в `L_shield`; резкие скачки коэффициентов → в `L_smooth`.
Shield реализуется и тестируется **до** актора — как чистая функция над `(поправки, coeffs, runtime_state)`,
с юнит-тестами на каждое правило.

## 10. Neural PID Gain Scheduler (NPGS) — актор + критик (`agent/gain_scheduler.py`)

> **АКТУАЛИЗАЦИЯ (реализовано):** сеть переведена с *мультипликативных поправок α* на **предсказание
> абсолютных коэффициентов** PID — это разблокировало SFT-подогрев (BC к эталонным пресетам; PPO с нуля
> сходится медленно). Отображение выхода: `gain_i = ref_i·exp(s_i·tanh(z_i))` вокруг геом-середины семейства
> пресетов (`agent/gain_space.py`), веса `1+tanh(z)`; головы L/R тормоза/реверса теперь **независимы** (17
> выходов = ACTION_DIM); старт ≈ DEFAULT через bias голов; per-output `log_std` (равномерный мульт. шаг).
> Безопасность/ТЗ: Shield держит **пресет сценария** как якорь границ/fallback (не глобальный DEFAULT).
> SFT-конвейер — `agent/pretrain.py` + `runtime/{capture,pretrain}.py` (захват классики в X-Plane → BC,
> анти-копирование шумом gain-признаков), тёплый старт в PPO — `TrainConfig.init_from`. Ниже — исходный
> дизайн α-поправок (сохранён для истории; действующая архитектура — абсолютные коэффициенты).

**Название и почему не «PIDNN».** Классический *PIDNN* встраивает передаточную функцию PID в саму сеть —
сеть **становится** регулятором. ТЗ это запрещает: PID остаётся классическим plant'ом, а сеть лишь
**предсказывает мультипликативные поправки** к его коэффициентам (+ веса каналов). Поэтому это не PIDNN, а
**планировщик коэффициентов**. Имя модуля — **Neural PID Gain Scheduler (NPGS)** (синонимы: Neural PID
Auto-Tuner, Adaptive Neural PID Gain Predictor). Термин «gain scheduling» точен: сеть подбирает gain'ы по
режиму движения, а иерархический контекст фаз (ниже) делает это буквально расписанием по фазе.
*(`PIDNN.mmd` пока сохраняет легаси-имя; диаграмма обновится при реализации, Этап 4.)*

**Вход и выбор рекуррентности.** На такте подаётся **окно из `T` кадров** наблюдения (§5): тензор
`(B, T, 56)`. При 20 Гц `T = 16` ≈ 0.8 с истории (настраивается; больше `T` — длиннее память динамики шин/
гидравлики/инерции). Используем **оконный** рекуррентный проход (RNN пересчитывает окно каждый такт), а не
stateful-RNN с переносом скрытого состояния: это убирает известные баги PPO + hidden-state/усечённого BPTT,
даёт детерминизм для поставки и совпадает с кольцевым буфером `rollout_env`. *(Альтернатива — stateful-GRU,
стиль SB3 RecurrentPPO; на Этапе 4 буфер отдаёт окно как последовательность `(T, 56)`, а не плоский вектор.)*

**Общий энкодер (актор и критик делят его):**
1. **Input LayerNorm** (по признакам каждого кадра) — обучаемая аффинная нормировка поверх фиксированной
   нормировки obs; детерминирована (deploy-safe). *(RunningMeanStd как в SB3 — альтернатива, но ломает
   «фиксированный контракт» и требует заморозки статистик; т.к. obs уже нормирован — не рекомендуется.)*
2. **Feature Encoder** (time-distributed, общие веса по `T`): `Linear(56→256) → GELU → Linear(256→256) →
   GELU` + residual. → `(B, T, 256)`. Извлекает признаки до рекуррентного слоя.
3. **Temporal Encoder:** `GRU(256, 2 слоя)` (рекоменд. — проще/быстрее для реалтайма) [альт. `LSTM(256)×2` —
   лучше долгая память]. → последовательность `(B, T, 256)` + финальное скрытое.
4. **Temporal Attention:** `MultiHeadAttention(256, heads=4)` self-attention по времени + residual, затем
   attention-пулинг в один вектор `(B, 256)` — сеть взвешивает информативные кадры (переходный процесс
   касания и т.п.).
5. **Shared trunk:** `Linear(256→256) → GELU → Linear(256→128) → GELU` + skip-connection. →
   `z_shared ∈ (B, 128)` — общее латентное пространство актора и критика.

**Иерархия: Context Fusion (фаза движения).** От `z_shared` — ветвь `Linear(128→128) → GELU` → **контекст
фазы** `c ∈ (B, 128)`. Опц. вспомогательная задача: предсказание фазы (касание / скоростной пробег / средняя
скорость / руление / остановка) с auxiliary-loss (метки из `groundspeed`) — улучшает представление и
интерпретируемость. Именно `c` реализует переключение вклада голов по режиму: ниже 60 узлов головы реверса
почти перестают менять коэффициенты, головы тормозов становятся основными — это соответствует логике
контура и обучается легче полностью независимых голов. Каждая голова получает `[z_shared ⊕ c]` (256).

**Головы (актор).** Тело головы: `Linear(256→64) → GELU → Linear(64→32) → GELU → Linear(32→out)`.
- **Heading head** → α для `runway_center_pid` (3).
- **Brake head** → α тормозов (3, **дублируется** на `brake_l`/`brake_r` — симметричное общее торможение,
  §1 принцип 2; асимметрия идёт через `w_lat`/дифф. торможение латерального канала, а не через L/R gain'ы).
- **Reverse head** → α реверса (3, дублируется на `rev_l`/`rev_r`).
- **Channel-weights head** → `(w_lon, w_lat)` (2).

Итого **11 выходов политики** (3+3+3+2), детерминированно разворачиваются в **17-мерный `Corrections`**
(порядок = `REGULATOR_ORDER` + веса, совпадает с `action.py`/`Shield`). *(Опция: независимые L/R (17
выходов) и расширение до 6 на регулятор — + `anti_windup/integral_decay/der_filter` — зарезервировано, §5.)*

**Ограничение выхода → стабильность PPO.** Поправки ограничены гладко:
`α = 1 + Δα_max · tanh(z)` (identity при `z=0`; `Δα_max = 0.5` → `α ∈ [0.5, 1.5]` = уровень 1 Shield);
веса `w = 1 + tanh(z) → [0, 2]`. Голова mean инициализируется малым orthogonal-gain (~0.01) → на старте
`α ≈ 1` (≈ классика), что согласуется с identity-инвариантом (§1) и «инициализацией около identity»
(критерий Этапа 4).

**Политика (PPO).** tanh-squashed Gaussian: голова даёт `mean` (pre-activation), `log_std` — обучаемый
вектор, state-independent, init ≈ −0.5. Сэмпл `z ~ N(mean, σ)`, `α = bound(z)`, log-prob с tanh-поправкой
`−Σ log(1 − tanh²(z))`. Greedy (поставка) = `mean`, без сэмплинга.

**Критик — голова, не отдельная сеть.** От `z_shared`: `Linear(128→64) → GELU → Linear(64→1) → V(s)`.
Общий энкодер с актором; баланс через `vf_coef` (если сходимость нестабильна — отдельный критик).

**Лучшие практики (RL + управление ЛА):**
- Orthogonal-init (gain `√2` скрытые, `0.01` mean-голова, `1.0` value-голова) → старт ≈ классика.
- GELU + LayerNorm, residual/skip; **dropout ≤ 0.05 или нет** (в RL ухудшает политику).
- GAE(λ), нормировка advantage, value-clipping, малый entropy-бонус, grad-clip (max-norm 0.5),
  annealing LR, KL-early-stop, нормировка return.
- **Реалтайм < 50 мс/такт**: `GRU-256×2` + малые головы — единицы мс на CPU; ~1–2 M параметров (доминирует GRU).
- **Детерминизм поставки:** greedy = mean, LayerNorm детерминирован, веса + нормировка заморожены (§14).
- Смещённость к identity + Shield ниже по потоку → безопасный старт обучения и поставки.

**Два ЛА (A330-300 в X-Plane ↔ МС-21 «Иркут» на стенде).** Обучение только на X-Plane; перенос на МС-21 —
за счёт domain randomization (робастность) + Shield (безопасность), т.к. на МС-21 сеть не обучалась.
Контекст фазы помогает обобщению по режимам. Опц. **зарезервировать вход plant-conditioning** (эмбеддинг
типа ЛА) под возможное дообучение на данных стенда. Sim-to-bench gap — ключевой риск (§17 п.3).

## 11. PPO + многокомпонентный loss (`agent/ppo.py`, `runtime/train.py`)

Диаграмма, блок 6. `L = L_ppo + λ_s·L_smooth + λ_p·L_phys + λ_sh·L_shield`.

- **L_ppo** — стандартный clipped surrogate + GAE-advantage от критика.
- **L_smooth** — штраф за рывки управления и резкие скачки коэффициентов PID (плавность gain).
- **L_phys** — *опционально*, включается вместе с обсервером (§12): штраф за нефизичный residual.
  До появления обсервера `λ_p = 0`, но слагаемое и хук в loss уже присутствуют.
- **L_shield** — штраф за вмешательство Shield (по уровням).
- Оптимизатор — AdamW.
- **Покомпонентный критик / reward** (описание, разд. V): отдельно оцениваются боковое отклонение,
  ошибка скорости, рывки, активация Shield, срыв по курсу, нестабильность. Reward-термы и их веса — в
  `envs/reward.py`, пороги — из `config/requirements.py`.
- Логирование: TensorBoard/CSV по каждому терму, манифест запуска (seed, сценарии, гиперпараметры).

**Выбор библиотеки — согласовано: компактный собственный PPO на PyTorch** (стиль CleanRL) — из-за
нестандартного действия (мультипликативные поправки + веса каналов), multi-head политики и
кастомного многокомпонентного loss с хуками Shield/Phys. Stable-Baselines3 остаётся запасным вариантом
через custom policy, но кастомизация loss в нём тяжелее.

## 12. (Опция) PINN / PD-PINN Observer (`agent/observer.py`)

Пока **не реализуем**, но проектируем швы. Диаграмма, блок 2.

- **Назначение:** восстановление скрытых параметров динамики → выполнение диагностических требований ТЗ
  5.1.3 (μ, боковой ветер, аквапланирование) и обогащение состояния актора:
  - μ (коэффициент сцепления ВПП);
  - эффективное торможение → коэффициент эффективности тормозов (из фактического замедления и команды);
  - степень деградации эффективности руления (NWS) — по фактическому повороту при заданной команде;
  - боковая компонента ветра;
  - задержка/инерционность динамики (τ) — с учётом инерции ЛА, люфта, проскальзывания шин, гидравлики.
- **Обучение отдельно:** классический System Identification при взаимодействии с симулятором, возможно без
  PPO. Архитектура: история состояний → Temporal Encoder (GRU/LSTM) → Dense → Physics Head → `μ̂, ŵ, brake_eff, nws_eff, τ̂`.
- **Точки интеграции, резервируемые сейчас:**
  1. Слоты обсервера в Observation Space (§5, помечены «Опция») — заранее в схеме вектора.
  2. Тип `ObserverEstimate` и метод `SimInterface`/env, возвращающий оценки (сначала заглушка нулями/
     ground-truth из сценария).
  3. Слагаемое `L_phys` и его хук в loss (§11), `λ_p = 0` до включения.
- За нефизичное поведение обсервер начисляет штраф в `L_phys`.

### Этап 3 — Shield (защитный контур) — ✅ ВЫПОЛНЕНО (в коде)

**Цель:** детерминированный контур безопасности между актором и PID, не обучается, всегда активен, с
fallback. См. §9. Построен **до** актора (автономный, покрыт юнит-тестами) — по §9.

**Итог (`agent/shield.py`):**
- ✅ Структуры: `Corrections` (поправки `α×15` + веса `w_lon/w_lat`; `from_vector`/`to_vector`,
  `ACTION_DIM=17` — готово под выход сети), `RuntimeState`, `ShieldReport` (флаги уровней, `l_shield`,
  `l_smooth`, список сработавших правил — для интерпретации вклада, ТЗ 5.1.5), `ShieldConfig` (все границы/
  веса — настраиваемые).
- ✅ Уровень 1: clip `α` (`[0.5,1.5]`) и весов каналов (`[0,2]`).
- ✅ Уровень 2: применение поправок к базовым gain'ам → hard bounds (мультипликативные вокруг базы),
  консистентность (неотрицательность), rate-limit между тактами, OOD-детектор (сырой выход за широким
  диапазоном → fallback).
- ✅ Уровень 3: реверс <60 узлов (снятие), чрезмерно резкое торможение (rate-limit), срыв по курсу
  (soft-штраф / hard-fallback с защёлкой на следующий такт).
- ✅ Fallback: OOD/грубый срыв → поправки заменяются на identity (классика).
- ✅ Штрафы: активация уровня → `L_shield`; скачки коэффициентов/команд → `L_smooth`.
- ✅ Хелперы интеграции: `base_gains_from_pids` / `apply_gains_to_pids` (мост к `ControllingSystem.pids`).
- **Инвариант identity** (§1): `α=1`, веса=1 → эффективные gain'ы == базовым, Shield молчит (тест).

**Точки интеграции (Этап 4):** `guard_coefficients(corrections, base_gains)` — ДО `control_step`,
`guard_command(command, runtime_state)` — ПОСЛЕ, на итоговом `ControlsState`.

**Артефакты:** `agent/shield.py`, `agent/__init__.py`; `tests/test_shield.py` (15 тестов: все уровни/
правила, OOD-fallback, защёлка курса, rate-лимиты, инвариант identity, векторный round-trip, хелперы).
Всего в наборе — 65 тестов.

### Этап 4 — NPGS (Neural PID Gain Scheduler) + Critic + PPO — ✅ ВЫПОЛНЕНО (в коде)

**Цель:** обучаемый планировщик коэффициентов, генерирующий поправки и веса каналов; PPO с многокомпонентным
loss; обучение с domain randomization. **Полная архитектура — §10.**

**Итог:**
- ✅ `agent/gain_scheduler.py` — `NPGS` (~1.4 M параметров): общий энкодер `LayerNorm → FeatureEncoder(Dense×2
  +residual) → GRU(256)×2 → MultiHeadAttention(256,4)+residual → attention-пулинг → Shared trunk(Dense×2)+skip`
  → `z_shared(128)`; ветвь **Context Fusion** → `c(128)` (+ голова фазы движения под вспом. задачу); головы
  Heading/Brake/Reverse/Weights (**11** выходов → **17**-мерный `Corrections`; тормоза/реверс дублируются на
  L/R — симметрия, §1 п.2); выход `α=1+0.5·tanh(z)∈[0.5,1.5]`, `w=1+tanh(z)∈[0,2]`; политика — tanh-squashed
  Gaussian (обучаемый `log_std`); **голова критика** от `z_shared`. Orthogonal-init, mean-головы малым gain →
  старт `α≈1` (identity, §1). `save/load` — веса + конфиг + слепок нормировки одним артефактом (deploy).
- ✅ `agent/ppo.py` — `PPOTrainer`: rollout-буфер одного env, GAE(λ), clipped surrogate + value-clipping;
  `L = L_ppo + λs·L_smooth + λp·L_phys(=0) + λsh·L_shield(=0)`; AdamW, grad-clip 0.5, нормировка advantage,
  entropy-бонус, KL-early-stop, annealing LR; логирование по каждому терму в `history`. `L_smooth` —
  дифференцируемый штраф отклонения поправок от identity; недифференцируемые штрафы (рывки/Shield/курс) идут
  через **reward** (в advantage); `L_phys`/`L_shield` — швы с нулевым весом под обсервер (§12) и барьер Shield.
- ✅ `runtime/train.py` — цикл обучения на **общем** коннекторе X-Plane (среда+контур), curriculum по
  `difficulty` от `ScenarioGenerator`, CSV-лог + чекпоинты (веса+нормировка), **Shield в inference-пути**
  (`guard_coefficients` до `control_step`, `guard_command` после); `silence_control_console()` гасит
  потактовый `cprint` контура (иначе флуд на 20 Гц×5); `smoke_train(...)` — оффлайн-прогон без X-Plane.
- ✅ `rollout_env` — окно истории теперь **последовательность `(T, 56)`** (вход NPGS), а не плоский вектор.

**Критерий готовности:**
- ✅ Формы forward `(B,T,56) → (mean 11, value, phase)`; ограничение выхода — identity при `z=0` **бит-в-бит**
  == `IDENTITY_ACTION`; экстремальный вход всегда в band; greedy детерминирован; `evaluate_actions` совпадает
  с `get_action`; save/load round-trip; разворот действия == `Corrections`/`Shield` (симметрия L/R).
- ✅ Такт greedy-инференса ≈ **2.6 мс** (CPU) / ≈12 мс (CUDA) — с запасом < 50 мс.
- ✅ PPO сходится по механике: параметры обновляются, все термы loss конечны, `L_smooth` тянет к identity
  (оффлайн-прогон на скриптованной среде + юнит-тесты GAE/loss).
- ⏳ Рост reward и прохождение гейтов §2 «не хуже классики» — на реальном прогоне обучения в X-Plane
  (переходит в приёмку Этапа 5). Старт ≈ классика обеспечен (identity-инициализация + Shield).

**Артефакты:** `agent/{gain_scheduler,ppo}.py`, `runtime/train.py`; `tests/test_gain_scheduler.py` (11) +
`tests/test_ppo.py` (7): формы/identity/band/greedy/save-load, GAE, компоненты loss, сквозной PPO на
скриптованной среде, `(T,56)`-окно. Всего в наборе — **95 тестов** (torch-тесты через `importorskip`).
